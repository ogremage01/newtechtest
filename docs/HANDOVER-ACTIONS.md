# 인수인계 시 내가 할 일 (행동 가이드)

상황: **내가 이 프로젝트에서 손을 뗀다. 소스코드·Docker·SSH 접속을 전부 다음 관리자에게 넘겨야 한다.**

아래 순서대로 진행하면 된다. 각 항목을 끝낸 뒤 체크해 두자.

---

## 1. 넘길 목록 정리

넘겨야 하는 것을 먼저 적어 둔다.

| 구분 | 넘길 것 | 비고 |
|------|----------|------|
| 소스코드 | 이 저장소 전체 (또는 사용 중인 브랜치/폴더) | Git 저장소 또는 압축 파일 |
| Docker | 이미지 사용 방식 + 레지스트리 또는 이미지 파일 | 팀 레지스트리 계정 또는 tar |
| SSH | 서버 접속 수단 | 키/비밀번호 + 접속 정보 |

---

## 2. 소스코드 넘기기

### 2-1. Git 저장소를 쓰는 경우 (권장)

- [ ] **저장소가 내 개인 계정이면**  
  - 팀/조직 계정으로 **저장소 이전(transfer)** 하거나  
  - 팀 계정에 **새 저장소를 만들고 push**한 뒤, 그 URL을 넘긴다.  
  - 다음 관리자를 **Collaborator / 팀 멤버**로 추가해 권한을 준다.

- [ ] **이미 팀·조직 소유 저장소면**  
  - 다음 관리자를 저장소 멤버로 추가하고,  
  - 기본 브랜치와 사용 중인 브랜치 이름을 문서로 적어 넘긴다.

- [ ] **넘기는 사람이 할 일**  
  - 최신 코드 push 여부 확인  
  - `README.md`, `docs/OPERATIONS.md`, `docs/HANDOVER-ACTIONS.md`가 최신인지 확인  
  - 저장소 URL + 기본 브랜치 + “어디서 클론하면 되는지” 한 줄 안내를 적어서 전달

### 2-2. Git 없이 코드만 넘기는 경우

- [ ] 프로젝트 루트에서 불필요한 파일 제외 후 압축  
  - 예: `node_modules/`, `backend/build/`, `frontend/.next/`, `.env`(비밀 포함 시 제외)  
  - 압축 파일명에 날짜 넣기 (예: `newtechtest-20250302.zip`)

- [ ] 압축 파일을 **팀 공용 공간**(NAS, 공유 드라이브, 내부 전달 수단)에 올리거나, 다음 관리자에게 직접 전달

- [ ] “이 압축이 최신 소스다”라고 적어서 함께 전달

## 3. Docker 넘기기

### 3-1. 팀/조직 레지스트리를 쓰는 경우 (권장)

- [ ] **이미지를 팀 레지스트리에 push 해 둔 상태로 넘긴다.**  
  - 개인 Docker Hub가 아니라 **팀·회사 계정**의 레지스트리 사용 (Docker Hub Organization, GitHub CR, GitLab, 사내 레지스트리 등)

- [ ] **다음 관리자에게 전달할 내용** (비밀은 별도 보안 채널로):  
  - 레지스트리 주소 (예: `ghcr.io/회사명/newtechtest-app`)  
  - 이미지 이름·태그 (예: `newtechtest-app:latest`)  
  - **로그인 계정/토큰** → 팀 비밀 저장소에 등록해 두고, “비밀 저장소의 ○○ 항목 사용”이라고만 문서에 적기

- [ ] 서버나 CI에서 `docker pull` 할 때 쓰는 **계정/토큰**이 내 개인 계정이면,  
  **팀 계정으로 바꾸거나 팀 비밀 저장소에 등록한 뒤** 그 사용법만 문서로 넘긴다.

### 3-2. 레지스트리 없이 이미지 파일로 넘기는 경우

- [ ] 로컬에서 이미지 저장:  
  `docker save -o newtechtest-app.tar newtechtest-app:latest`  
  (필요하면 `| gzip` 으로 압축해 `.tar.gz` 로 전달)

- [ ] **이 tar(또는 tar.gz)를** 팀 공용 공간에 올리거나 다음 관리자에게 전달

- [ ] **함께 전달할 안내 문구 예시:**  
  “서버에 복사한 뒤 `docker load -i newtechtest-app.tar` 실행하면 이미지가 로드된다.  
   실행 방법은 `docs/OPERATIONS.md` 3절 참고.”

## 4. SSH 연결 넘기기

- [ ] **서버 접속 정보**를 정리해서 전달한다.  
  - 호스트(IP 또는 도메인)  
  - 포트 (기본 22가 아니면 명시)  
  - 접속 사용자명  
  - **인증 방식**: 비밀번호인지, SSH 키인지

- [ ] **비밀번호로 접속하는 경우**  
  - 비밀번호는 **문서에 직접 쓰지 말고**, 팀 비밀 저장소나 보안 채널로 전달  
  - 문서에는 “서버 비밀번호: ○○ 비밀 저장소의 ‘서버명’ 항목”처럼만 적기

- [ ] **SSH 키로 접속하는 경우**  
  - 다음 관리자에게 **전용 키를 새로 발급**하는 것을 권장 (내 키를 복사해 주기보다는)  
  - 새 관리자가 자신의 공개키를 서버 `~/.ssh/authorized_keys`에 등록하게 안내  
  - 이미 내 키를 넘겨야 한다면: 키 파일을 **암호화하거나 비밀 저장소로 전달**하고, 문서에는 “SSH 키: ○○ 경로/비밀저장소”만 적기

- [ ] **접속 후 할 일**을 한 줄로 적어 둔다.  
  예: “Docker 설치됨. 이미지는 `docker load` 또는 `docker pull` 후 `docker run` 또는 `docker compose`로 실행. 자세한 건 OPERATIONS.md 참고.”

## 6. 다음 관리자와 함께 할 일 (마지막 단계)

- [ ] **짧은 인수인계 미팅**  
  - 저장소(또는 압축) 위치, Docker 이미지 위치, SSH 접속 방법을 말로 한 번 훑어 준다  
  - 비밀/토큰은 미팅에서 말하지 말고, “비밀 저장소 ○○ 항목 확인해라”만 안내

- [ ] **다음 관리자가 직접 해 보게 하기**  
  - 저장소 클론(또는 압축 풀기)  
  - 로컬에서 `docker compose up -d app` 실행해 보기  
  - SSH로 서버 접속해 보기  
  - (이미지 사용 중이면) 서버에서 `docker pull` 또는 `docker load` 후 실행해 보기

- [ ] **질문 받을 채널/기간**  
  - “○일까지는 채팅/메일로 질문 받는다” 정도를 정해 두면 좋다

## 요약: 인수인계 시 내가 할 행동

1. **소스** → Git은 팀 저장소로 이전/공유 + 멤버 추가. Git 없으면 압축해서 전달.  
2. **Docker** → 팀 레지스트리에 push된 이미지 + 로그인 정보(비밀저장소) 전달. 또는 tar로 이미지 전달 + 로드 방법 안내.  
3. **SSH** → 서버 접속 정보 + 인증(비밀번호/키)는 비밀저장소·보안 채널로 전달하고, 문서에는 “보관 위치”만 적기.  
4. **문서** → README, OPERATIONS, 이 HANDOVER-ACTIONS를 최신으로 맞춘 뒤, 다음 관리자와 한 번 같이 따라 해 보기.

이 순서대로 진행하면 소스·Docker·SSH를 정리해서 넘길 수 있다.
